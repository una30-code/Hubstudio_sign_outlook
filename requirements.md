# 需求说明（Requirements）

> 本文档为当前阶段唯一需求来源；与实现冲突时以本文为准。

---

## 0. 总目标

通过配置驱动方式，完成以下分阶段流程：

1. phase-0：创建 Hubstudio 指纹浏览器环境
2. phase-1：生成 Outlook 注册所需用户信息
3. phase-2：根据 phase-0 产出的环境 ID 打开指纹浏览器，并执行 Outlook 注册页基础交互流程

---

## 1. 分阶段需求

### 1.1 phase-0 指纹浏览器环境准备

目标：通过配置文件创建可用的 Hubstudio 指纹浏览器环境，对创建环境的信息留档，以备后续使用。

#### 功能要求

- 必须从配置文件读取多条环境创建参数，不允许硬编码。
- 必填项至少包含：
  - 环境名称
  - IP 信息（代理或网络配置）
  - 指纹参数（浏览器基础指纹配置）
- 创建成功后，需返回可用于后续启动的环境标识（如 profile_id / 环境名称等）。
- 需对创建结果留档，至少包含：环境标识（如 `containerCode`）、环境名称、创建时间、region、执行状态（success/fail）。
- 留档数据需可被后续 phase 读取（用于 phase-2 启动与回溯）。

#### 验收标准

- 提供完整配置时，环境创建成功，返回 success = True。
- 缺少必填项时，创建失败并返回明确错误信息。
- 关键步骤有日志记录（开始创建、参数校验、创建结果）。
- 创建成功后存在可查询的留档记录（字段完整，可追溯到本次运行）。

### 1.2 phase-1 用户信息生成

目标：自动生成符合约束的用户信息，对用户信息也进行留档，用于 Outlook 注册流程。

#### 功能要求

- 姓名：美国风格姓名（名 + 姓）。
- 生日：年龄范围 18-55 岁（含边界）。
- 账号：基于美国姓名生成，并包含 5 位随机数字。
- 密码：长度 10 位，且必须同时包含数字、小写字母、大写字母。
- 需对生成用户信息留档，至少包含：姓名、生日、账号、生成时间、执行状态（success/fail）。
- 留档信息需支持后续 phase-2 读取与回溯（不要求明文密码写入日志）。

#### 验收标准

- 生成结果字段完整（姓名、生日、账号、密码）。
- 字段格式满足上述规则。
- 输出结构包含 success / message / data（或等价结构）。
- 生成成功后存在可查询留档记录，且记录与本次输出一致（敏感字段可脱敏存储）。

### 1.3 phase-2 Outlook 邮箱创建（基础流程）

目标：根据 **phase-0 生成的指定指纹浏览器环境 ID**（如 `containerCode` 或 Hubstudio 文档中的等价环境标识）打开指纹浏览器，在此基础上完成 Outlook 注册页自动化基础交互，输入phase-1 生成的用户信息，完成注册。

#### 功能要求

- **环境 ID 来源（配置驱动，禁止硬编码）**：必须使用与 phase-0 创建成功结果一致的**环境标识**；优先从 phase-0 **留档**（archive）读取，也允许通过 `.env` / 配置文件显式传入同一 ID 以便联调。
- **打开环境**：根据上述环境 ID，调用 Hubstudio 约定能力**打开/启动**对应该环境的指纹浏览器会话；实现上可对已有会话先 **best-effort 关闭再启动**，以保证 CDP 调试端口可用（详见 `design.md` §10）。
- **消费 phase-1 产物**：从 `logs/archive/phase1_user_profile_*.jsonl` **最近一条成功留档**读取账号、密码等，用于在注册页**录入**；无留档时 phase-2 应失败并提示先执行 phase-1。
- 自动打开 **Outlook 注册页面**（URL 可配置，见 §2）。
- 执行基础页面交互流程：
  - 页面加载等待与超时控制；**填表单步**等待可与整页导航超时分离（见 `PHASE2_FORM_TIMEOUT_MS` / `design.md` §10.3）
  - 将 phase-1 用户信息填入可见字段：邮箱/用户名、密码、**姓名与生日（人物身份）**，并在各步后点击「下一步/继续」等主按钮
  - 关键元素存在性检查（进入录入前须通过页面校验）

#### 验收标准

- 环境 ID 缺失、与 phase-0 留档不一致或 Hubstudio 无法识别时，失败并返回明确错误，流程不崩溃。
- 在有效配置与服务可用时：能按**指定环境 ID**成功打开指纹浏览器并进入可自动化状态。
- 可成功获取 **browser / context / page**（或 Playwright 等价对象）。
- 注册页 URL 正确且页面加载完成（在约定超时内）。
- 页面出现注册相关关键元素（标题、输入框、按钮等之一）。
- 在具备页面上下文时完成**信息录入与中间步骤提交**（至人物身份页提交后）；输出 **success = True**，并记录关键日志；失败时按 §3 尽量保留截图。
- **说明**：§1.3 所称「完成注册」指自动化完成**邮箱→密码→人物信息**等约定步骤；**人机验证、手机/邮箱验证码、后续风控页**等受 §4 约束，不保证绕过。

## 2. 输入与输出约定

### 输入

- Hubstudio 环境创建相关配置（phase-0，来源：配置文件 / `.env`）
- **phase-2**：**phase-0 产出的指纹浏览器环境 ID**（或可从留档解析得到该 ID 的引用，如 `archive_ref` / 最近一次成功记录的查询方式，由实现约定）
- Outlook 注册 URL（phase-2）
- **phase-1 用户信息留档**（phase-2 **必须**可读：默认从 `logs/archive/phase1_user_profile_*.jsonl` 最近成功记录读取；与 phase-1 输出字段一致）

### 输出

- success: bool
- message: str
- data: dict（可选，包含环境信息、用户信息、页面状态等）
- logs: 关键步骤日志
- screenshot_path: str（失败时必须输出或可定位）
- archive_path / archive_ref：留档文件路径或记录标识（至少二选一，见 phase-0/1 约定）

---

## 3. 非功能需求

### 稳定性

- 页面加载与关键步骤必须设置超时。
- 任一阶段失败时，流程不崩溃，返回可读错误信息。

### 可观测性

- 必须记录关键步骤日志与错误日志。
- 失败时必须截图，保存到项目根目录 `screenshots/`。
- 对外返回统一结果结构（success / message / error）。
- phase-0 与 phase-1 必须有可追溯留档（与 logs 区分：logs 为运行日志，archive 为业务结果记录）。

### 安全

- 不在代码中硬编码账号、密码、token、cookie 等敏感信息。
- 配置统一从 `.env` 或受控配置文件读取（`.env` 不入库）。

---

## 4. 范围外（当前版本不做）

- **不保证**完成 Microsoft 侧全部开户闭环（如验证码、手机验证、异常风控、CAPTCHA 等）；实现以当前可见表单项为准。
- 不做账号批量管理与生命周期管理。
- 不实现与需求无关的浏览器环境高级参数优化。

---

## 5. 成功定义

满足以下条件即视为成功：

1. phase-0：可通过配置成功创建 Hubstudio 环境。
2. phase-0：创建结果留档成功，可被后续 phase 使用。
3. phase-1：可稳定生成符合规则的用户信息。
4. phase-1：用户信息留档成功，可用于后续注册流程对接。
5. phase-2：可按 phase-0 环境 ID 打开指纹浏览器，打开 Outlook 注册页，关键元素校验通过，并按 phase-1 留档完成邮箱、密码与人物身份等步骤及中间提交（§4 所列风控步骤除外）。
6. 流程返回 success = True，且关键步骤有日志可追溯。

---

## 6. 失败定义

以下任一情况视为失败：

- 配置缺失或格式错误导致环境创建失败。
- 浏览器环境无法启动或无法连接；或 phase-2 使用的环境 ID 无效、与 phase-0 留档不一致、Hubstudio 无法打开对应环境。
- 页面无法打开、超时或 URL 不符合预期。
- 页面关键元素缺失，无法进入下一步。
- 抛出未处理异常，或返回结构不完整。
- phase-0 / phase-1 执行成功但未完成留档，或留档不可查询。
- phase-2 需要 phase-1 留档但缺失或不可读。
